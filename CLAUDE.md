## Commands

```powershell
# dev / build — both auto-mirror graphify-out/graph.json into public/graph/ via predev/prebuild
npm run dev
npm run build
npm run lint

# refresh the knowledge graph (AST-only, no LLM, $0)
npm run graph:sync     # = graphify update . && node scripts/sync-graph.mjs

# production preview
npm run start
```

There is no test runner configured. The single source of correctness is `npx tsc --noEmit` plus the Next.js dev compiler output.

## Big-picture architecture

The site is a single client-rendered Next.js 16 page (`app/page.tsx`) that composes sections top-to-bottom: **Loader → Nav → Hero → About → NeuralBrain → ObsidianGraph → Marquee → HorizontalScroll → Contact**. Two draggable overlay components, `Terminal` (Ctrl+`) and `PlaygroundWindow`, are toggled from Nav and live outside `<main>`.

Three different rendering engines are intentionally in play; reading one alone misses the picture:

- **react-three-fiber** drives `NeuralBrain` and 11 of the 12 Playground scenes (in `components/playground/scenes/`). All Playground scenes share one R3F `<Canvas>` and the camera helper at `components/playground/lib/scene-camera.tsx` (drei `OrbitControls` + `prefers-reduced-motion` gate).
- **postprocessing** (`@react-three/postprocessing`) — Bloom + ChromaticAberration via `components/playground/lib/effects.tsx`. Per-scene intensity is declared in the `DEMOS` array inside `components/PlaygroundCanvas.tsx`; setting `bloom: 0` disables it entirely.
- **PlayCanvas** (`components/playground/PlayCanvasDemo.tsx`) owns its own `<canvas>`. When the user opens the PLAYCANVAS tab, `PlaygroundCanvas` unmounts the R3F Canvas entirely and mounts the PlayCanvas `pc.Application` — two WebGL engines never coexist. The runtime is dynamic-imported so the ~400KB cost only happens on first activation.

Heavy components are loaded via `next/dynamic` with `ssr: false` (NeuralBrain, ObsidianGraph, PlayCanvasDemo) to avoid SSR/WebGL errors.

## Knowledge graph integration

This repo is itself a graphify corpus. `graphify-out/graph.json` is the AST-derived knowledge graph (~169 nodes, 25 communities), regenerated by `graphify update .`. Two parts of the running site consume it:

1. `components/ObsidianGraph.tsx` — fetches `/graph/graph.json` (mirrored from `graphify-out/` by `scripts/sync-graph.mjs`) and renders a force-directed canvas with wikilink-style neighbor navigation. The mirror runs automatically on `predev`/`prebuild`; refresh manually with `npm run graph:sync`.
2. `notes/` — Obsidian wiki pages (`Home.md`, `index.md`, `Community_*.md`) generated by `Deploy/build_wiki.py` from the same `graph.json`. The `.obsidian/` config at the repo root makes the project itself a vault.

`.codex/hooks.json` registers `graphify hook-check` as a PreToolUse Bash hook (auto-rebuilds the graph after code changes). The optional git post-commit auto-rebuild is **not** installed by default; opt in with `graphify hook install`.

## AGENTS.md priority

`AGENTS.md` puts the `## graphify` block above the `<!-- BEGIN:nextjs-agent-rules -->` block on purpose — the knowledge graph is the primary map of the codebase, and Next.js conventions are the secondary constraint. The Next.js block is load-bearing: this is Next.js 16, which has breaking changes vs older training data. Before editing anything Next-specific, read `node_modules/next/dist/docs/`. For cross-module "how does X relate to Y" questions, prefer `graphify query "<q>"`, `graphify path "<A>" "<B>"`, or `graphify explain "<concept>"` over grep.

## Backups and operation logs

- `Deploy/snapshot-*/` — point-in-time source snapshots taken before risky operations (one full tree per snapshot, excluding `node_modules`/`.next`/`.git`). Roll back by copying contents back over the project root.
- `Deploy/build_wiki.py` — regenerator for `notes/` from the current `graph.json`.
- `memory/*.md` — operation logs (what changed, why, rollback recipe). These double as breadcrumbs for the next Claude session and should be appended to (not edited) when shipping a non-trivial change.

## Conventions

- Path alias `@/*` resolves to the repo root (see `tsconfig.json`).
- The whole site uses `cursor: none` (set in `globals.css`); the visible cursor is rendered by `components/CustomCursor.tsx`, which scales its ring when hovering anything that's an `a`, `button`, or has a `data-hover` attribute. New interactive elements should add `data-hover="true"`.
- Three font CSS vars are wired in `app/layout.tsx`: `--font-playfair` (display serif), `--font-inter` (body), `--font-mono` (UI / terminal / monospace blocks). Reference them via `fontFamily: 'var(--font-...)'` rather than naming the families directly.
- Palette: gold `#c9a96e`, gold-light `#e8d5b0`, dark `#0a0a0a`, surface `#111`. Defined as Tailwind 4 `@theme` tokens in `globals.css`.
- WebGL warnings about `THREE.Clock` deprecation and the occasional `WebGLRenderer: Context Lost` predate active work and originate inside Three/drei; they're informational, not actionable.
